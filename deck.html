Step 01 · 提出问题
第 1 步「提出问题」—— 整个流程的起点

整个流程的第一步不是调研，而是提出问题——在做具体市场调研前，先把要解决的问题想清楚。不然你根本无从下手：爬谁的数据、验证什么，全没方向。具体做法是打磨一个可检验的问题假设，围绕三个追问反复磨：

追问 01
谁有这个问题？
这是否构成一个市场？
用户画像够不够具体？人群够不够大？
追问 02
已经有人在
解决它了吗？
如果有，他们做得怎么样？效果如何？
追问 03
方案要做到
什么程度才算真解决？
我的想法做到了那个程度吗？
终极问题
三个追问最终汇聚成一个问题 · 这值得构建吗？
答案是肯定的，才往下走。否则——回去打磨问题，或换个方向。
Step 01 · 举例 · 可检验的问题假设
几个例子 · 把"可检验的问题假设"理解透
例 01 · 模糊 vs 可检验
✕ 模糊版
"现在人都很孤独，我想做个 AI 陪伴产品"——谁孤独、什么场景、多频繁都没说清，没法验证。
✓ 可检验版
"独居、社交圈窄的年轻上班族，深夜情绪低落时缺一个能即时倾诉、又记得住自己的对象"——人群 / 场景 / 频次都具体，才能拿数据去验证。
例 02 · Airbnb 先验证再做
问题假设：大型展会期间酒店订满、价格暴涨，有人愿意花钱住进别人家的空房。
创始人在设计大会期间自己出租气垫床验证假设——没做产品，先用最土的方式确认问题真实。
例 03 · Uber 先验证再做
最早根本没做 App——先用一个电话号码：用户打过来，Uber 的人手动找个司机再回电。
用最土的方式确认"打车难"是真问题，再谈产品。

MRD · 市场需求文档
Market Requirements Document
MRD —— 用户到底要什么

MRD 只关注市场与用户，暂不涉及怎么做、能否盈利。一份 MRD 通常包含 4 块：

Block 01
数据质量评估
先说清这批数据能分析什么、不能分析什么。后续结论才不会失真。
Block 02
痛点聚类
从几千条评论归纳出 3-5 个主要方向，每类配真实原声佐证。
Block 03
需求优先级
P0 / P1 / P2 标注，按频率、情绪强度、是否影响留存判断。
Block 04
竞品格局
用户在抱怨现有方案哪里不好，我们的机会窗口在哪。
示例
MRD 示例 · 减脂健身 App

爬取数千条 Keep、薄荷健康的真实评论后，可能聚出三类痛点：

P0 必须做
坚持不下来
评论最多 · 情绪最强 · 直接影响留存
P1 很想要
不知道该吃什么
很想要但非必需
P2 锦上添花
在家练怕动作不标准
提及较少
竞品格局 · Keep
"跟练"做得很好，但用户抱怨"什么都要会员""课程太多挑花眼"。
机会窗口 → 轻、不挑花眼、专注帮用户坚持
BRD · 商业需求文档
Business Requirements Document
BRD —— 这事值不值得做

有需求不等于值得做。BRD 负责算账与决策，通常包含 4 块：

Block 01
继承 MRD
直接读取 MRD 的市场结论往下接，不重复分析、不重复提问。
Block 02
商业可行性
谁付费 · 怎么定价 · 成本结构 · 最小闭环（粗算单位经济）。
Block 03
明确结论
值得做 / 有条件地做 / 建议暂缓 三选一，必须明确表态，不和稀泥。
Block 04
关键风险
列出依赖的关键假设，指出哪个一旦不成立结论就翻转。
示例
BRD 示例 · 共享充电宝

付费者 = 临时手机没电又离不开手机的人；定价如每小时三元；成本含机柜、商家分成、运维补货。这门生意成立与否，卡在"单台日均借用次数"这个数字上：

单台测算
平均每次借用收 5 元，每天借 8 次 = 40 元，扣除分成与折旧 → 约数月回本。
但若每天仅借 2 次 → 永远无法回本。
明确结论
值得做，但有条件：仅在日均借用高于阈值的点位铺设。
关键风险
电池续航普遍提升、用户自带充电宝增多 → 整体借用率下滑，根基便动摇。
PRD · 产品需求文档
Product Requirements Document
PRD —— 具体做什么、怎么做

方向确定、值得做之后，才进入产品本身。PRD 要细到能直接交给 AI 编码，通常包含 4 块：

Block 01
继承 BRD
读取 BRD 的商业判断与 P0/P1 功能列表往下接。
Block 02
MVP 范围
V1 最多 3 个核心功能；附"不做清单"，且不做清单比要做的更长更显眼。
Block 03
页面 & 组件
每页有什么、每个组件做什么（交互、状态、边界），写到可直接执行。
Block 04
设计交接
产品调性、色彩方向、参考竞品，供设计与开发使用。
示例
PRD 示例 · 待办清单 App

这类产品最易陷入"什么都想加"。PRD 会把第一版收敛为三件事，并写一份更长的"不做清单"：

要做 · 三件事
  • 新增任务
  • 标记完成
  • 删除任务
不做清单（更长更显眼）
  • 不做提醒
  • 不做多人协作
  • 不做标签
  • 不做日历
  • 不做云同步
  • 不做撤销
组件示例 · 任务行
左侧空心圆圈点击后变打勾、文字加删除线置灰；列表为空显示引导文案而非白屏；右侧删除按钮直接删除（本版不做撤销）。
串行 vs 一次生成
为什么三份文档要串行
看市场 → 决定做不做 → 才定怎么做
判断点清晰
MRD 让你看清市场，BRD 让你决定做不做，PRD 才开始定怎么做。跳过任何一步都可能整体跑偏。
接力棒交接
每份文档末尾有结构化交接区，下一份自动读取，不重复提问、不丢信息。
人始终拍板
AI 干执行（爬数据、做分析、写文档、写代码）；判断永远由人来定。

先搞懂
三个词

前端 · 后端 · 数据库——经典三层结构。
前端 = 餐厅大堂
前端 —— 运行在用户浏览器里的代码
职责
把界面画出来（渲染 UI）· 响应点击和输入 · 把请求发给后端。
餐厅比喻
大堂
桌椅、菜单、点菜的地方——客人看得见的那部分。
本项目里 = 登录页、聊天界面、记忆档案页。
技术栈
React + Tailwind + Next.js
React 写界面、Tailwind 调样式，整个项目跑在 Next.js 框架上。
后端 = 餐厅后厨
后端 —— 运行在服务器上的代码
职责
业务逻辑 · 身份验证 · 调用第三方服务（大模型 API）· 读写数据库。前后端靠接口（API）通信。
餐厅比喻
后厨
客人看不到、活儿都在这儿干。你发一句话 → 后端拼"人设+记忆" → 送给大模型 → 拿回复传回前端。
为什么必须有
大模型密钥只能放服务器端，绝不能写进前端——前端代码用户能看到，等于把钥匙挂门口。
数据库 = 仓库冰箱
数据库 —— 数据的持久化存储
职责
不存进去，内存里的东西一刷新就没了。关系型数据库按"表"存：一行一条记录、一列一个字段，用 SQL 增删改查。
餐厅比喻
仓库和冰箱
食材、账本存这儿，关了门也不丢。本项目里 = 账号、聊天记录、长期记忆。
技术栈
Supabase（底层 PostgreSQL）
业界主流的关系型数据库，顺手把登录注册也一起做了。
